I approached the AI-assisted coding assignment with a deliberate intention to remain the owner and director of the coding project. Surprisingly, my goal was harder to maintain as the project progressed and the more AI-suggested code ran successfully and accurately. The biggest benefit I experienced from AI-assisted coding was time savings; instead of spending time looking up syntax or library patterns, I used AI to handle the more routine coding tasks and was able to focus on the overall design of the cloud-kitchen simulation. Using AI was especially helpful for laying out initial scaffolding: defining function signatures, suggesting test structures, and proposing a reasonable module organization very quickly. Additionally, AI helped make up for my lack of experience writing and using unit tests.

AI was also a useful teammate in its ability to explain code. For the original instructor-provided scripts, I asked for a plain language summary of what the code did and was able to ask follow-up questions for clarifications, saving me from reading every single line while still giving me enough understanding to evaluate whether the behavior matched the assignment’s requirements.

However, the assignment showed me how risky AI-assisted coding can be, with the most obvious risk being the temptation to let the AI handle every piece of the assignment. As the AI completes tasks successfully, it can be easy to accept a plausible-looking answer and move on, especially when the code runs or the tests pass. However, complete dependency on AI-assisted coding will lead to a shallow understanding of the codebase, limited ability to debug when things go wrong, and potential errors that only become apparent in edge cases. I noticed that if I wasn’t careful, I would skim the AI’s changes instead of really reviewing them.

One concrete feature that helped mitigate this was GitHub Copilot’s workflow of presenting code additions and deletions for explicit approval. Having to accept or reject each change nudged me to analyze code deltas more carefully, think about side effects, and occasionally ask the AI to revise or simplify a solution. The additional step of approval  made the collaboration feel more like pair programming than blind code generation.

I also noticed differences between tools. I originally started the assignment with Codex and then transitioned to GitHub Copilot, due to my enterprise switching tools after Codex went down due to an issue in their FedRamp environment. Both are GPT-based, but they responded differently. Codex often picked up patterns in my prompts more quickly, especially when I was using the four AI prompt templates. Copilot, in contrast, tended to need more explicit, step-by-step direction, but still integrated well with my development flow, tests, and repository, pushing me to be more clear about my own intent.

Overall, I see AI-assisted coding as a powerful way to increase efficiency and reduce tedious work but also as a potential for cutting intellectual corners. The key is to treat the AI as a tool, not an author: ask it to propose code, explanations, and tests, but retain responsibility for understanding and validating the result. For this project, the most successful moments were when I used AI to handle mechanical tasks while I focused on design, correctness, and alignment with the rubric.
